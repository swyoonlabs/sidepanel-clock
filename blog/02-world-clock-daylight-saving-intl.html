A world clock without the time-zone headaches (and yes, Daylight Saving too)

A world clock sounds trivial until you remember Daylight Saving Time. New York is UTC−5 in winter and UTC−4 in summer. Sydney does the opposite, because the seasons are flipped. India sits on a half-hour offset; Nepal on a 45-minute one. The moment you try to store fixed offsets, you've signed up to maintain the world's DST rules forever. In this post I'll show how I avoided all of that by leaning on one browser API: Intl.

The key idea: names, not numbers

The single most important decision is to never store an offset like UTC+9. Instead, store an IANA time-zone name such as Asia/Seoul or America/New_York. These names map to the full history and future of a region's rules — including when DST starts and stops — inside the browser's built-in time-zone database. 

const TIMEZONES = [
  ["seoul",   "Asia/Seoul"],
  ["tokyo",   "Asia/Tokyo"],
  ["london",  "Europe/London"],
  ["newyork", "America/New_York"],
  ["sydney",  "Australia/Sydney"],
  // …20 cities total
];

To render the current time for a zone, I hand that name straight to Intl.DateTimeFormat: 

const timeStr = new Intl.DateTimeFormat(uiLang, {
  timeZone: "America/New_York",
  hour: "2-digit",
  minute: "2-digit",
  hour12: true,
}).format(new Date());
💡 There is no Daylight Saving code in the app. Because the zone is named, the browser decides whether DST applies for today's date and converts accordingly. If a country changes its rules, a browser update fixes it — not an app update.

The offset label: "−3h 30m vs local"

Showing the time is easy. The genuinely fiddly part is the little label that says how far ahead or behind the city is relative to you. My first version parsed a localized date string and rounded to whole hours — which quietly broke for India (+5:30) and Nepal (+5:45). The fix was to compute the offset in minutes, using formatToParts to read a zone's wall-clock time reliably: 

// How many minutes is this zone ahead of UTC right now?
// (DST- and half-hour-offset aware)
function zoneOffsetMinutes(zone, now) {
  const parts = Object.fromEntries(
    new Intl.DateTimeFormat("en-US", {
      timeZone: zone, hour12: false,
      year: "numeric", month: "2-digit", day: "2-digit",
      hour: "2-digit", minute: "2-digit", second: "2-digit",
    })
      .formatToParts(now)
      .map((p) => [p.type, p.value])
  );
  const hour = parts.hour === "24" ? 0 : +parts.hour; // midnight quirk
  const asUTC = Date.UTC(
    +parts.year, +parts.month - 1, +parts.day,
    hour, +parts.minute, +parts.second
  );
  return Math.round((asUTC - now.getTime()) / 60000);
}

The trick: format the instant as if it were in the target zone, then read those components back as though they were UTC. The gap between that and the real timestamp is the zone's offset — in minutes, so half- and quarter-hour zones survive. Note the "24" guard: some environments format midnight as hour 24 instead of 0.

The difference from your own zone is then just a subtraction. JavaScript's Date.getTimezoneOffset() returns minutes to add to reach UTC (so it's negative east of UTC), which makes the local part a one-liner: 

function localOffsetLabel(zone, now) {
  const diff = zoneOffsetMinutes(zone, now) + now.getTimezoneOffset();
  if (diff === 0) return "Same as local";
  const sign = diff > 0 ? "+" : "-";
  const abs = Math.abs(diff);
  const h = Math.floor(abs / 60), m = abs % 60;
  return m ? `${sign}${h}h ${m}m vs local` : `${sign}${h}h vs local`;
}

Spot-checking a few cases with local time set to Seoul (UTC+9) shows why minutes matter:

  • Delhi (+5:30) → −3h 30m in both summer and winter (India has no DST).
  • New York → −13h in summer (EDT), −14h in winter (EST). DST, handled for free.
  • Sydney → +1h in the northern summer, +2h in winter — the southern-hemisphere flip, also for free.

A tiny touch: day or night?

Because I already had reliable per-zone hours, I added a small badge showing whether it's dawn, day, dusk, or night in each city — surprisingly useful for "is it a reasonable hour to message them?" 

function dayNightGlyph(zone, now) {
  const h = zoneHour(zone, now); // built on the same Intl call
  if (h >= 5 && h < 8)  return "🌅"; // dawn
  if (h >= 8 && h < 18) return "☀️"; // day
  if (h >= 18 && h < 20) return "🌆"; // dusk
  return "🌙";                           // night
}

Bonus: city names in the user's language

The same "let the platform do it" philosophy applies to city names. Rather than translate them myself, each city has a message key, and the browser's i18n system returns Seoul / 서울 / ソウル / 首尔 based on the UI language. The list even re-sorts alphabetically in that language.

Takeaways

  • Store IANA zone names, never fixed offsets — DST comes for free.
  • Compute offsets in minutes via formatToParts to support +5:30 / +5:45 zones.
  • Guard the hour "24" midnight edge case.
  • Lean on Intl for formatting and localized names — less code, fewer bugs.
